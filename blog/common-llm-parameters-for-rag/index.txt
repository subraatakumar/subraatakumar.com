1b:T3723,<p class="blog-md-p">RAG quality is not only about model choice.</p>
<p class="blog-md-p">In practice, your results depend heavily on <strong>how you tune parameters</strong> across both retrieval and generation.
The wrong knobs can cause hallucinations, missed context, slow responses, and higher cost.
The right knobs make your assistant grounded, fast, and reliable.</p>
<p class="blog-md-p">This guide is provider-neutral and focuses on the most practical parameters to tune first.</p>
<hr class="blog-md-hr" />
<h2 id="a-retrieval-parameters-what-context-you-fetch" class="blog-md-h2">A) Retrieval Parameters (What Context You Fetch)</h2>
<h3 id="1-topk" class="blog-md-h3">1) <code class="blog-md-inline-code">topK</code></h3>
<p class="blog-md-p">How many chunks/documents you retrieve before generation.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Too low: you may miss relevant evidence.</li>
<li class="blog-md-li">Too high: you flood context with noise and increase latency/cost.</li>
</ul>
<p class="blog-md-p">Practical default: <code class="blog-md-inline-code">topK</code> between <code class="blog-md-inline-code">4</code> and <code class="blog-md-inline-code">8</code> for most QA cases.</p>
<h3 id="2-scorethreshold-or-minsimilarity" class="blog-md-h3">2) <code class="blog-md-inline-code">score_threshold</code> (or <code class="blog-md-inline-code">min_similarity</code>)</h3>
<p class="blog-md-p">Minimum relevance score required to keep a retrieved chunk.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Higher threshold: cleaner but fewer chunks.</li>
<li class="blog-md-li">Lower threshold: more coverage but more irrelevant context.</li>
</ul>
<p class="blog-md-p">Use this to improve precision when retrieval returns weak matches.</p>
<h3 id="3-reranktopn" class="blog-md-h3">3) <code class="blog-md-inline-code">rerank_top_n</code></h3>
<p class="blog-md-p">Number of retrieved items sent to a reranker model to improve ordering.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Useful when semantic search returns mixed quality in top results.</li>
<li class="blog-md-li">Often better than blindly increasing <code class="blog-md-inline-code">topK</code>.</li>
</ul>
<p class="blog-md-p">Example: retrieve 20, rerank, keep top 5.</p>
<h3 id="4-chunksize" class="blog-md-h3">4) <code class="blog-md-inline-code">chunk_size</code></h3>
<p class="blog-md-p">How large each chunk is when indexing documents.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Small chunks: precise retrieval but risk losing broader context.</li>
<li class="blog-md-li">Large chunks: richer context but lower precision and more token usage.</li>
</ul>
<p class="blog-md-p">Typical starting range: <code class="blog-md-inline-code">300-800</code> tokens.</p>
<h3 id="5-chunkoverlap" class="blog-md-h3">5) <code class="blog-md-inline-code">chunk_overlap</code></h3>
<p class="blog-md-p">How much neighboring chunk text repeats.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Helps preserve meaning across chunk boundaries.</li>
<li class="blog-md-li">Too much overlap increases index size and near-duplicate retrieval.</li>
</ul>
<p class="blog-md-p">Typical starting range: <code class="blog-md-inline-code">50-120</code> tokens.</p>
<h3 id="6-context-budget" class="blog-md-h3">6) Context Budget</h3>
<p class="blog-md-p">How many retrieved tokens you actually include in the final prompt.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Even good retrieval can fail if you overfill context.</li>
<li class="blog-md-li">Reserve space for system instructions and output tokens.</li>
</ul>
<p class="blog-md-p">Treat context as a budget, not an unlimited buffer.</p>
<hr class="blog-md-hr" />
<h2 id="b-generation-parameters-how-the-model-responds" class="blog-md-h2">B) Generation Parameters (How The Model Responds)</h2>
<h3 id="1-temperature" class="blog-md-h3">1) <code class="blog-md-inline-code">temperature</code></h3>
<p class="blog-md-p">Controls randomness.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Lower (<code class="blog-md-inline-code">0.0-0.3</code>): more deterministic, good for factual RAG.</li>
<li class="blog-md-li">Higher (<code class="blog-md-inline-code">0.6+</code>): more creative, but higher hallucination risk.</li>
</ul>
<h3 id="2-topp-nucleus-sampling" class="blog-md-h3">2) <code class="blog-md-inline-code">top_p</code> (nucleus sampling)</h3>
<p class="blog-md-p">Chooses from the smallest set of tokens whose cumulative probability reaches <code class="blog-md-inline-code">p</code>.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Lower <code class="blog-md-inline-code">top_p</code>: more conservative output.</li>
<li class="blog-md-li">Higher <code class="blog-md-inline-code">top_p</code>: broader token exploration.</li>
</ul>
<p class="blog-md-p">Common practical range: <code class="blog-md-inline-code">0.8-1.0</code>.</p>
<h3 id="3-topk-if-supported" class="blog-md-h3">3) <code class="blog-md-inline-code">top_k</code> (if supported)</h3>
<p class="blog-md-p">Restricts next-token sampling to the top <code class="blog-md-inline-code">k</code> candidates.</p>
<p class="blog-md-p">Important: this is a <strong>generation</strong> parameter, different from retrieval <code class="blog-md-inline-code">topK</code>.</p>
<h3 id="4-maxoutputtokens-or-maxtokens" class="blog-md-h3">4) <code class="blog-md-inline-code">max_output_tokens</code> (or <code class="blog-md-inline-code">max_tokens</code>)</h3>
<p class="blog-md-p">Caps output length.</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Too low: answers get cut off.</li>
<li class="blog-md-li">Too high: unnecessary cost and slower responses.</li>
</ul>
<p class="blog-md-p">Set this based on expected answer style (short QA vs long explanation).</p>
<h3 id="5-stopsequences" class="blog-md-h3">5) <code class="blog-md-inline-code">stop_sequences</code></h3>
<p class="blog-md-p">Defines where generation should stop.</p>
<p class="blog-md-p">Useful for:</p>
<ul class="blog-md-ul">
<li class="blog-md-li">forcing concise answers,</li>
<li class="blog-md-li">separating answer/citation sections,</li>
<li class="blog-md-li">reducing verbose drift.</li>
</ul>

<h2 id="d-debug-checklist-symptom-likely-cause-first-fix" class="blog-md-h2">D) Debug Checklist (Symptom -> Likely Cause -> First Fix)</h2>
<ol class="blog-md-ol">
<li class="blog-md-li"><strong>Hallucinations despite RAG</strong> -> noisy retrieval -> increase <code class="blog-md-inline-code">score_threshold</code> and reduce <code class="blog-md-inline-code">topK</code>.</li>
<li class="blog-md-li"><strong>Correct answer exists but is missed</strong> -> low recall -> raise <code class="blog-md-inline-code">topK</code> or improve chunking.</li>
<li class="blog-md-li"><strong>Answer is too generic</strong> -> weak grounding -> tighten prompt to require citations and reduce temperature.</li>
<li class="blog-md-li"><strong>Response is slow</strong> -> too much context -> lower <code class="blog-md-inline-code">topK</code>, reduce chunk size, apply reranking.</li>
<li class="blog-md-li"><strong>Answer is cut mid-sentence</strong> -> output cap too low -> increase <code class="blog-md-inline-code">max_output_tokens</code>.</li>
<li class="blog-md-li"><strong>Repeated near-duplicate evidence</strong> -> overlap too high -> reduce <code class="blog-md-inline-code">chunk_overlap</code>.</li>
<li class="blog-md-li"><strong>Inconsistent answers run-to-run</strong> -> high randomness -> lower <code class="blog-md-inline-code">temperature</code> and keep <code class="blog-md-inline-code">top_p</code> stable.</li>
</ol>

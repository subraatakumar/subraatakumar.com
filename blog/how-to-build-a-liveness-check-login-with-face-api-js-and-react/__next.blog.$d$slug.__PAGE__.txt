<p class="blog-md-p"><em>"What if your login page could tell the difference between a real person and a photo of one?"</em></p>

<p class="blog-md-p">That's the idea we're going to explore today. We'll look at how to build an <strong>anti-spoofing face login</strong> — a login system that asks users to do something like <em>"blink twice"</em> or <em>"turn your head left"</em> before letting them in. <h2 id="the-problem-with-regular-face-login" class="blog-md-h2">The Problem with Regular Face Login</h2>
<p class="blog-md-p">Imagine you build a face login. A user looks at their webcam, the app recognises them, and they're in. Sounds secure, right?</p>
<p class="blog-md-p">Not quite. What if someone just <strong>holds up a photo</strong> of the registered user? A basic face recognition system won't know the difference. It sees a face, matches it, and logs in the attacker. This is called a <strong>spoofing attack</strong>.</p>
<p class="blog-md-p">The fix is a <strong>liveness check</strong> — asking the user to perform a real-time action that a photo simply cannot replicate. Things like:</p>
<ul class="blog-md-ul">
<li class="blog-md-li">Blink twice</li>
<li class="blog-md-li">Smile</li>
<li class="blog-md-li">Turn your head to the left</li>
<li class="blog-md-li">Open your mouth</li>
</ul>
<p class="blog-md-p">This is what makes face login actually secure. And the good news? We can implement all of this <strong>entirely in the browser</strong>, for free, using open-source tools.</p>
<hr class="blog-md-hr" />
<h2 id="meet-face-apijs" class="blog-md-h2">Meet face-api.js</h2>
<p class="blog-md-p"><a href="https://github.com/justadudewhohacks/face-api.js/" class="blog-md-link"><code class="blog-md-inline-code">face-api.js</code></a> is an open-source JavaScript library for face detection and recognition. It runs directly in the browser (or Node.js) and is built on top of <strong>TensorFlow.js</strong> — which means it uses your device's GPU for fast, smooth performance.</p>
<p class="blog-md-p">The best part for privacy-conscious apps? <strong>Everything runs on the client side.</strong> The user's face data never leaves their device.</p>
<p class="blog-md-p">Here's a quick overview of what it can do:</p>

<p class="blog-md-p">For our anti-spoofing login, the <strong>landmark detection</strong> is the star. Those 68 points tell us exactly where the eyes, eyelids, and mouth corners are — which is exactly what we need to detect a blink or a smile.</p>

<h2 id="how-does-face-recognition-actually-work" class="blog-md-h2">How Does Face Recognition Actually Work?</h2>
<p class="blog-md-p">Let's demystify the process before we write any code.</p>
<h3 id="step-1-register-a-user" class="blog-md-h3">Step 1 — Register a User</h3>
<p class="blog-md-p">When a user signs up, your app:</p>
<ol class="blog-md-ol">
<li class="blog-md-li">Opens their webcam</li>
<li class="blog-md-li">Detects their face using <code class="blog-md-inline-code">face-api.js</code></li>
<li class="blog-md-li">Generates a <strong>face descriptor</strong> — a 128-number array (think of it as a mathematical fingerprint of that face)</li>
<li class="blog-md-li">Saves this descriptor (in the browser, localStorage, or a database)</li>
</ol>
<h3 id="step-2-login-attempt" class="blog-md-h3">Step 2 — Login Attempt</h3>
<p class="blog-md-p">When the same user tries to log in:</p>
<ol class="blog-md-ol">
<li class="blog-md-li">The webcam opens again</li>
<li class="blog-md-li">A new face descriptor is generated from the live feed</li>
<li class="blog-md-li">The app compares the new descriptor against all saved ones using <strong>Euclidean distance</strong> (how "far apart" two descriptors are)</li>
<li class="blog-md-li">If the distance is small enough (below a threshold, usually <code class="blog-md-inline-code">0.6</code>), it's a match</li>
</ol>
<h3 id="step-3-the-liveness-check-anti-spoofing" class="blog-md-h3">Step 3 — The Liveness Check (Anti-Spoofing)</h3>
<p class="blog-md-p">This is the extra layer. Before or after recognition, your app:</p>
<ol class="blog-md-ol">
<li class="blog-md-li">Shows a random challenge — <em>"Please blink twice"</em></li>
<li class="blog-md-li">Watches the landmark data from the video stream in real time</li>
<li class="blog-md-li">Checks if the eyes actually closed and opened (or the mouth opened for a smile)</li>
<li class="blog-md-li">Only approves the login if the challenge is passed</li>
</ol>
<p class="blog-md-p">A printed photo cannot blink. A video playing on a screen <em>could</em> fool a basic version, but adding randomised challenges makes even that impractical.</p>
<hr class="blog-md-hr" />
<h2 id="the-models-behind-the-magic" class="blog-md-h2">The Models Behind the Magic</h2>
<p class="blog-md-p"><code class="blog-md-inline-code">face-api.js</code> ships with several pre-trained neural network models. You load the ones you need:</p>
<p class="blog-md-p"><strong>SSD Mobilenet V1</strong> — The default, high-accuracy detector. Great when precision matters.</p>
<p class="blog-md-p"><strong>Tiny Face Detector</strong> — Roughly 190 KB. Built for real-time performance on mobile or low-powered devices. This is usually your go-to for a live video stream.</p>
<p class="blog-md-p"><strong>FaceLandmark68Net</strong> — Detects 68 key points on the face. This is what enables liveness detection.</p>
<p class="blog-md-p"><strong>FaceRecognitionNet</strong> — A ResNet-34-like architecture that generates the 128-number face descriptor used for identity matching.</p>
<p class="blog-md-p"><strong>FaceExpressionNet</strong> — Detects emotions. Useful if your liveness challenge is <em>"please smile."</em></p>
<p class="blog-md-p">You load them like this:</p>

        <div class="blog-code-head">
          <span class="blog-code-lang">JAVASCRIPT</span>

        </div>
        <pre class="blog-md-pre"><code class="language-javascript">await faceapi.nets.tinyFaceDetector.loadFromUri('/models');
await faceapi.nets.faceLandmark68Net.loadFromUri('/models');
await faceapi.nets.faceRecognitionNet.loadFromUri('/models');
await faceapi.nets.faceExpressionNet.loadFromUri('/models');

They're split into small ~4 MB chunks so the browser can download them in parallel and start working quickly.</p>

<h2 id="detecting-a-blink-the-core-of-anti-spoofing" class="blog-md-h2">Detecting a Blink — The Core of Anti-Spoofing</h2>
<p class="blog-md-p">So how do we actually detect a blink? The <code class="blog-md-inline-code">faceLandmark68Net</code> model gives us the positions of all 68 points on a face. Eyes occupy specific indices in that array.</p>
<p class="blog-md-p">A common technique is called the <strong>Eye Aspect Ratio (EAR)</strong>:</p>

      <div class="blog-code-wrap">
        <div class="blog-code-head">
          <span></span>
          <button data-copy class="blog-copy-btn">Copy</button>
        </div>
        <pre class="blog-md-pre"><code>       |p2 - p6| + |p3 - p5|
EAR = ─────────────────────────
            2 * |p1 - p4|
</code></pre>
      </div>
    
<p class="blog-md-p">When the eye is open, EAR is around <code class="blog-md-inline-code">0.3</code>. When it closes (a blink), EAR drops close to <code class="blog-md-inline-code">0</code>. By watching this value frame by frame on the video stream, you can reliably detect blinks.</p>
<p class="blog-md-p">Here's a simplified implementation:</p>

      <div class="blog-code-wrap">
        <div class="blog-code-head">
          <span class="blog-code-lang">JAVASCRIPT</span>
          <button data-copy class="blog-copy-btn">Copy</button>
        </div>
        <pre class="blog-md-pre"><code class="language-javascript">function getEAR(eyePoints) {
  const vertical1 = Math.abs(eyePoints[1].y - eyePoints[5].y);
  const vertical2 = Math.abs(eyePoints[2].y - eyePoints[4].y);
  const horizontal = Math.abs(eyePoints[0].x - eyePoints[3].x);
  return (vertical1 + vertical2) / (2.0 * horizontal);
}

// In your video loop:
const detections = await faceapi
  .detectSingleFace(video, new faceapi.TinyFaceDetectorOptions())
  .withFaceLandmarks();

if (detections) {
  const landmarks = detections.landmarks;
  const leftEye = landmarks.getLeftEye();
  const rightEye = landmarks.getRightEye();
  
  const leftEAR = getEAR(leftEye);
  const rightEAR = getEAR(rightEye);
  const avgEAR = (leftEAR + rightEAR) / 2;

  if (avgEAR &#x3C; 0.25) {
    // Eye is closed — a blink is happening!
    blinkCount++;
  }
}
</code></pre>
      </div>
    
<p class="blog-md-p">A full liveness check waits for <code class="blog-md-inline-code">blinkCount >= 2</code> within a time window (say, 5 seconds) before approving the session.</p>

<h2 id="extending-this-into-a-fullstack-project" class="blog-md-h2">Extending This into a Fullstack Project</h2>
<p class="blog-md-p">The prototype is client-side only, which is fine for learning — but a real production system needs a backend. Here's how you'd extend it:</p>
<h3 id="on-the-frontend-react" class="blog-md-h3">On the Frontend (React)</h3>
<ol class="blog-md-ol">
<li class="blog-md-li">Capture the user's face descriptor using <code class="blog-md-inline-code">face-api.js</code></li>
<li class="blog-md-li">Run the liveness challenge (e.g., blink detection)</li>
<li class="blog-md-li">Only if the challenge passes, send the descriptor to the server</li>
</ol>
<h3 id="on-the-backend-nodejs-any-language" class="blog-md-h3">On the Backend (Node.js / any language)</h3>
<ol class="blog-md-ol">
<li class="blog-md-li">During <strong>registration</strong>: receive the face descriptor, associate it with a user account, store it in a database</li>
<li class="blog-md-li">During <strong>login</strong>: receive the live descriptor, query all saved descriptors for that user (or look them up by username first), compute distances, and return success/failure</li>
</ol>

      <div class="blog-code-wrap">
        <div class="blog-code-head">
          <span class="blog-code-lang">JAVASCRIPT</span>
          <button data-copy class="blog-copy-btn">Copy</button>
        </div>
        <pre class="blog-md-pre"><code class="language-javascript">// Pseudocode — backend login check
app.post('/login', async (req, res) => {
  const { username, liveDescriptor, livenessVerified } = req.body;

  if (!livenessVerified) {
    return res.status(401).json({ error: 'Liveness check failed' });
  }

  const user = await db.findUser(username);
  const distance = faceapi.euclideanDistance(liveDescriptor, user.faceDescriptor);

  if (distance &#x3C; 0.6) {
    return res.json({ success: true, token: generateJWT(user) });
  }

  return res.status(401).json({ error: 'Face not recognised' });
});
</code></pre>
      </div>
    
<h3 id="security-considerations" class="blog-md-h3">Security Considerations</h3>
<ul class="blog-md-ul">
<li class="blog-md-li"><strong>Never trust the client</strong> to tell the server that liveness passed. Re-verify server-side where possible, or at minimum use a signed token from the liveness check step.</li>
<li class="blog-md-li"><strong>Encrypt stored face descriptors.</strong> They're mathematical data, not photos, but they're still biometric.</li>
<li class="blog-md-li"><strong>Add rate limiting.</strong> Prevent brute-force attempts against the face matcher.</li>
<li class="blog-md-li"><strong>Use HTTPS always.</strong> Webcam access requires a secure context.</li>
</ul>

<h2 id="real-world-use-cases-of-this-technology" class="blog-md-h2">Real-World Use Cases of This Technology</h2>
<p class="blog-md-p">This isn't just a cool side project — the underlying tech powers some serious real-world systems:</p>
<p class="blog-md-p"><strong>Corporate Attendance</strong> — Touchless check-in portals that verify employees are physically present (not dialling in from home while showing a photo at the scanner).</p>
<p class="blog-md-p"><strong>Two-Factor Auth</strong> — Replacing the second factor in 2FA with a face + liveness check instead of an SMS code.</p>
<p class="blog-md-p"><strong>KYC (Know Your Customer)</strong> — Fintech and banking apps verifying a user's identity during onboarding.</p>
<p class="blog-md-p"><strong>Exam Proctoring</strong> — Ensuring the person who sat the registration test is the same person taking the exam.</p>
<p class="blog-md-p"><strong>Accessibility</strong> — Hands-free navigation for users with motor disabilities who can log in or control interfaces using head movements or expressions.</p>
<hr class="blog-md-hr" />
<h2 id="understanding-model-formats-a-quick-primer" class="blog-md-h2">Understanding Model Formats (A Quick Primer)</h2>
<p class="blog-md-p">If you dig into the <code class="blog-md-inline-code">/models</code> folder of a face-api.js project, you'll notice files like <code class="blog-md-inline-code">tiny_face_detector_model-weights_manifest.json</code> and several <code class="blog-md-inline-code">.bin</code> files. This is the <strong>TensorFlow.js format</strong>.</p>
<p class="blog-md-p">Here's a quick cheat sheet for the formats you'll encounter as you explore AI:</p>
<div class="blog-md-table-wrap"><table class="blog-md-table">
<thead>
<tr>
<th class="blog-md-th">Format</th>
<th class="blog-md-th">Extension</th>
<th class="blog-md-th">Best For</th>

<tr>
<td class="blog-md-td">TensorFlow.js</td>
<td class="blog-md-td"><code class="blog-md-inline-code">.json</code> + <code class="blog-md-inline-code">.bin</code></td>
<td class="blog-md-td">Running models in the browser</td>
</tr>
<tr>
<td class="blog-md-td">TFLite</td>
<td class="blog-md-td"><code class="blog-md-inline-code">.tflite</code></td>
<td class="blog-md-td">Mobile apps (Android / iOS)</td>

<p class="blog-md-p">Think of it like image formats: a RAW file is great for editing in Photoshop (like a <code class="blog-md-inline-code">.pt</code> file), but you need a JPEG for a website (like <code class="blog-md-inline-code">.json</code> + <code class="blog-md-inline-code">.bin</code> shards) because it's compressed and browser-ready.</p>

<h2 id="what-to-build-next" class="blog-md-h2">What to Build Next</h2>
<p class="blog-md-p">Once you've cloned the prototype and got it running, here are some ideas to level it up:</p>
<ol class="blog-md-ol">
<li class="blog-md-li">
<p class="blog-md-p"><strong>Add a randomised liveness challenge</strong> — Instead of always asking to blink, randomly pick from: blink, smile, turn left, turn right. This makes replay attacks much harder.</p>
</li>
<li class="blog-md-li">
<p class="blog-md-p"><strong>Persist descriptors to a backend</strong> — Connect a Node.js + MongoDB (or PostgreSQL) backend so registrations survive a page refresh.</p>
</li>
<li class="blog-md-li">
<p class="blog-md-p"><strong>Add a fallback login</strong> — Face login is great, but always give users a password fallback in case their camera fails.</p>
</li>
<li class="blog-md-li">
<p class="blog-md-p"><strong>Try it in React Native</strong> — TensorFlow Lite models can run on mobile via the <code class="blog-md-inline-code">@tensorflow/tfjs-react-native</code> package, bringing the same ideas to your native app.</p>
</li>
<li class="blog-md-li">
<p class="blog-md-p"><strong>Audit for edge cases</strong> — Test with glasses, different lighting, hats. Real-world faces are messy, and your threshold tuning matters.</p>
</li>
</ol>

<h2 id="wrapping-up" class="blog-md-h2">Wrapping Up</h2>
<p class="blog-md-p">Face recognition in the browser is no longer a futuristic concept — it's a practical, accessible tool that any React developer can start using today. <code class="blog-md-inline-code">face-api.js</code> abstracts away all the neural network complexity, leaving you to focus on building the experience.</p>
<p class="blog-md-p">The prototype at <a href="https://subraatakumar.github.io/face-recognition-authentication-system-prototype/" class="blog-md-link">subraatakumar.github.io/face-recognition-authentication-system-prototype</a> gives you a solid working foundation. From there, adding the liveness check is the key step that takes it from a demo to something genuinely production-worthy.</p>
<p class="blog-md-p">If you build on top of this or have questions, feel free to open an issue on the repo. Happy building! 🚀</p>

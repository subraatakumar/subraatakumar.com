<h2 id="45-repository-pattern" class="d180-md-h2">4.5 Repository Pattern</h2>
<p class="d180-md-p"><img src="/images/day4_repository_layer.png" class="rounded-md shadow-sm mx-auto block" alt="image.png"></p>
<h3 id="real-world-analogy" class="d180-md-h3">Real-World Analogy</h3>
<p class="d180-md-p">To understand the Repository Layer, imagine a large warehouse that stores thousands of goods. The warehouse is organized with shelves, codes, storage rules, and inventory systems. It is complex inside.</p>
<p class="d180-md-p">Now imagine the staff working in the office. When they need an item, They do not search shelves, decode storage labels, or manage inventory records.</p>
<p class="d180-md-p">Instead, they approach the <strong>warehouse clerk</strong>.</p>
<p class="d180-md-p">The warehouse clerk knows:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Where items are stored</li>
<li class="d180-md-li">How inventory is organized</li>
<li class="d180-md-li">How goods are retrieved</li>
<li class="d180-md-li">How returns are handled</li>
</ul>
<p class="d180-md-p">The office staff simply say, “Bring item X.”</p>
<p class="d180-md-p">The clerk retrieves it and hands it over.</p>
<p class="d180-md-p">In this analogy:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">The <strong>warehouse</strong> represents the database.</li>
<li class="d180-md-li">The <strong>warehouse clerk</strong> represents the repository.</li>
<li class="d180-md-li">The <strong>office staff</strong> represent the rest of the application.</li>
</ul>
<p class="d180-md-p">The repository hides the complexity of storage. It provides a clean and controlled way to access data. The application does not need to know how the database is structured internally — it only communicates with the repository.</p>
<p class="d180-md-p">This improves clarity, reduces duplication, and prevents different parts of the application from interacting with storage in inconsistent ways.</p>

<h3 id="repository-pattern-trade-offs-and-justification" class="d180-md-h3">Repository Pattern: Trade-offs and Justification</h3>
<p class="d180-md-p">Some developers argue that introducing a repository layer can add unnecessary complexity, especially in small projects. Modern ORMs often already provide abstraction over database operations, making an additional repository seem redundant or even a potential performance overhead. In simple CRUD applications, wrapping an ORM with another layer may result in boilerplate code without delivering meaningful architectural benefits.</p>
<p class="d180-md-p">However, in the context of Offline Life Archive, the repository layer is not merely an ORM wrapper — it enforces architectural boundaries. It prevents direct database access from UI components, isolates business logic, supports encryption control, and keeps the system extraction-ready for future evolution. The slight increase in structural code is a deliberate trade-off for long-term maintainability, modularity, and security integrity.</p>
<hr class="d180-md-hr" />
<h2 id="46-service-layer" class="d180-md-h2">4.6 Service Layer</h2>
<h2 id="why-the-warehouse-clerk-is-not-enough" class="d180-md-h2">Why the Warehouse Clerk Is Not Enough</h2>
<p class="d180-md-p">In the previous section, we saw how the warehouse clerk (Repository) protects access to the warehouse (Database). Office staff do not walk into storage directly — they request items through the clerk. This keeps storage organized and controlled.</p>
<p class="d180-md-p">However, imagine the office staff now need more than simple item retrieval.</p>
<p class="d180-md-p">Suppose they ask:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">“Can we ship 500 units today?”</li>
<li class="d180-md-li">“Is this customer eligible for priority dispatch?”</li>
<li class="d180-md-li">“Do we have enough stock across multiple sections?”</li>
<li class="d180-md-li">“If one item is unavailable, should we cancel the entire order?”</li>
</ul>
<p class="d180-md-p">The warehouse clerk cannot answer these questions. The clerk only knows how to retrieve items from shelves.</p>
<p class="d180-md-p">This is where the <strong>Warehouse Manager</strong> becomes essential.</p>
<hr class="d180-md-hr" />
<h2 id="the-role-of-the-warehouse-manager" class="d180-md-h2">The Role of the Warehouse Manager</h2>
<p class="d180-md-p">The warehouse manager does not personally fetch goods from shelves. Instead, the manager:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Applies company rules</li>
<li class="d180-md-li">Checks policies and limits</li>
<li class="d180-md-li">Validates requests</li>
<li class="d180-md-li">Coordinates multiple item retrievals</li>
<li class="d180-md-li">Decides whether an operation should proceed</li>
</ul>
<p class="d180-md-p">Only after making those decisions does the manager instruct the clerk to retrieve items.</p>
<p class="d180-md-p">In our architecture:</p>
<ul class="d180-md-ul">
<li class="d180-md-li"><strong>Warehouse</strong> → Database</li>
<li class="d180-md-li"><strong>Warehouse Clerk</strong> → Repository</li>
<li class="d180-md-li"><strong>Warehouse Manager</strong> → Service Layer</li>
<li class="d180-md-li"><strong>Office Staff</strong> → UI</li>
</ul>
<p class="d180-md-p">The manager ensures business rules are followed.</p>
<p class="d180-md-p">The clerk ensures data is retrieved correctly.</p>
<p class="d180-md-p">Without a manager, staff would rely only on the clerk. Storage would remain organized, but business logic would become inconsistent and scattered.</p>
<hr class="d180-md-hr" />
<h2 id="architectural-insight" class="d180-md-h2">Architectural Insight</h2>
<p class="d180-md-p">Repository protects storage access.</p>
<p class="d180-md-p">Service protects business rules.</p>
<p class="d180-md-p">You need both for a scalable system.</p>
<p class="d180-md-p">This layered separation aligns directly with the architectural philosophy defined earlier — keeping business logic isolated and preventing cross-contamination between layers.</p>
<p class="d180-md-p">That is why the Service Layer exists — even when the Repository Layer is already in place.</p>
<h3 id="461-purpose" class="d180-md-h3">4.6.1 Purpose</h3>
<p class="d180-md-p">The service layer contains business logic.</p>
<p class="d180-md-p">It does <strong>not</strong>:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Render UI</li>
<li class="d180-md-li">Directly initialize Realm</li>
</ul>
<p class="d180-md-p">It does:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Validate data</li>
<li class="d180-md-li">Transform models</li>
<li class="d180-md-li">Coordinate repository calls</li>
</ul>
<p class="d180-md-p">Location:</p>

      <div class="relative my-6 code-block">
        <div class="absolute right-3 top-3 flex items-center gap-2 z-50 pointer-events-auto">
          
          <button data-copy class="copy-btn d180-copy-btn">Copy</button>
        </div>
        <pre class="d180-code-pre line-numbers"><code>modules/archive/archive.service.ts
</code></pre>
      </div>
    
<p class="d180-md-p">Example flow:</p>

      <div class="relative my-6 code-block">
        <div class="absolute right-3 top-3 flex items-center gap-2 z-50 pointer-events-auto">
          
          <button data-copy class="copy-btn d180-copy-btn">Copy</button>
        </div>
        <pre class="d180-code-pre line-numbers"><code>UI → Service → Repository → Realm
</code></pre>
      </div>
    
<p class="d180-md-p">This ensures UI remains presentation-only.</p>
<hr class="d180-md-hr" />
<h2 id="47-layer-communication-flow" class="d180-md-h2">4.7 Layer Communication Flow</h2>
<p class="d180-md-p">With infrastructure implemented, the flow is now:</p>

      <div class="relative my-6 code-block">
        <div class="absolute right-3 top-3 flex items-center gap-2 z-50 pointer-events-auto">
          
          <button data-copy class="copy-btn d180-copy-btn">Copy</button>
        </div>
        <pre class="d180-code-pre line-numbers"><code>Screen (UI)
   ↓
Service (Business Logic)
   ↓
Repository (Data Access)
   ↓
Encrypted Realm (Data Layer)
</code></pre>
      </div>
    
<p class="d180-md-p">Each layer has a single responsibility.</p>
<p class="d180-md-p">No cross-contamination.</p>
<hr class="d180-md-hr" />
<h2 id="48-what-we-intentionally-did-not-implement" class="d180-md-h2">4.8 What We Intentionally Did Not Implement</h2>
<p class="d180-md-p">To preserve architectural clarity, we did not:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Add global state management</li>
<li class="d180-md-li">Introduce Redux or similar libraries</li>
<li class="d180-md-li">Implement theme toggling UI</li>
<li class="d180-md-li">Build feature screens</li>
<li class="d180-md-li">Implement sharing logic</li>
</ul>
<p class="d180-md-p">Complexity is introduced only when required.</p>
<hr class="d180-md-hr" />
<h2 id="49-security-considerations" class="d180-md-h2">4.9 Security Considerations</h2>
<p class="d180-md-p">Security is divided into two independent layers:</p>
<ol class="d180-md-ol">
<li class="d180-md-li"><strong>Data Encryption</strong>
<ul class="d180-md-ul">
<li class="d180-md-li">Protects stored data</li>
<li class="d180-md-li">Implemented via encrypted Realm</li>
</ul>
</li>
<li class="d180-md-li"><strong>Session Lock (Future Chapter)</strong>
<ul class="d180-md-ul">
<li class="d180-md-li">Protects access while app is open</li>
<li class="d180-md-li">Implemented as overlay (not navigation route)</li>
</ul>
</li>
</ol>
<p class="d180-md-p">Separation ensures flexibility and maintainability.</p>
<hr class="d180-md-hr" />
<h2 id="410-outcome-of-this-chapter" class="d180-md-h2">4.10 Outcome of This Chapter</h2>
<p class="d180-md-p">At the end of this chapter, the application can:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Launch successfully</li>
<li class="d180-md-li">Generate and securely store an encryption key</li>
<li class="d180-md-li">Initialize an encrypted Realm database</li>
<li class="d180-md-li">Persist data via repository</li>
<li class="d180-md-li">Maintain strict architectural boundaries</li>
</ul>
<p class="d180-md-p">No UI has been built yet.</p>
<p class="d180-md-p">However, the system foundation is complete.</p>
<hr class="d180-md-hr" />
<h2 id="411-key-takeaways" class="d180-md-h2">4.11 Key Takeaways</h2>
<ul class="d180-md-ul">
<li class="d180-md-li">Infrastructure precedes features.</li>
<li class="d180-md-li">Encryption is not an optional add-on.</li>
<li class="d180-md-li">Repository pattern isolates database logic.</li>
<li class="d180-md-li">Service layer isolates business logic.</li>
<li class="d180-md-li">UI must remain free of data-layer dependencies.</li>
</ul>
<p class="d180-md-p">This approach ensures:</p>
<ul class="d180-md-ul">
<li class="d180-md-li">Reduced refactor risk</li>
<li class="d180-md-li">Clear separation of concerns</li>
<li class="d180-md-li">Long-term scalability</li>
<li class="d180-md-li">Extractable core architecture</li>
</ul>
